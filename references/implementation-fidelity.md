# Implementation Fidelity

## Principle

Implement the accepted product behavior and visual direction as one contract. The active product or brand route owns the user mission, task priority, content truth, and composition; the `DirectionContract` owns the selected visual thesis; optional Modifier deltas refine only their declared fields and never create a secondary route hierarchy.

Implementation is not a styling pass. Preserve both the Functional delta and the visible decisions that make the surface specific, then prove them through the target project.

## Functional delta

Record the behavior that becomes real in the target project. A visually faithful surface with missing transitions, false data, or inaccessible controls is not an implementation.

An implementation-authorized new UI does not require a pre-accepted concept. It requires a selected or explicitly inference-authorized `DirectionContract`, a target project, and a complete Functional delta before the first slice.

For accepted-concept work, stop if the concept is unavailable. For every Implement route, stop if the target project or any required Functional delta is unavailable. Stop inside the selected Implement / Surface / Modifier window. Use the four-owner exception only when the selected route is exactly Implement / Product / Image + Reference-led for an accepted image concept. Every other Implement route keeps its selected Surface and Modifier owners within the normal window or uses sequential windows. Record Asset truth and Verify needs as named gaps. When missing media materially constrains proof, claims, accessibility, or composition, open Asset truth after the current window closes even when no file exists. Open Verify only after the required verifiable input or environment exists.

| Functional delta | Required record | Acceptance evidence |
| --- | --- | --- |
| Workflow | Name the actor, trigger, user-visible transition, next state, and recovery path. | Run the primary path and capture behavioral evidence for the observed transition. |
| Data | Name the source of truth, read and write path, freshness rule, and honest fallback. | Exercise representative data and record the observed result at its evidence locator. |
| Permissions | Name the authorization boundary, allowed actor, denied state, and escalation path. | Verify both allowed and denied behavior without inventing access. |
| Error recovery | Name the failure signal, retained input, retry or alternate path, and recovered state. | Trigger the applicable failure and observe recovery behavior. |
| Irreversible actions | Name the consequence, confirmation or approval, audit event, and rollback or explicit no-rollback rule. | Exercise a safe equivalent or record why the irreversible path cannot be run. |
| Accessibility | Name the keyboard path, focus behavior, semantic relationship, applicable pointer or touch target, and reduced-motion behavior. | Inspect semantics and run the keyboard, focus, target-input, and motion checks applicable to the declared device scope. |

## Direction lock

Treat the accepted `DirectionContract` and approved Modifier deltas as named decisions, not general inspiration.

- **Locked:** website archetype, route scope, ordered content beats, region geometry, interaction states, target-device scope, user mission, first and second read, dominant silhouette, composition family, experience architecture, page rhythm and continuity, content hierarchy, palette roles, type roles, material language, signature detail, and any accepted proof relationship.
- **Adaptable:** exact token values, component internals, breakpoint values, stacking mechanics, accessible contrast corrections, performance substitutions, and real data plumbing.
- A Modifier may lock a local relationship only inside its ownership boundary. It does not change the route owner or unrelated contract fields.
- If engineering constraints require a locked decision to change, stop that slice and record the change before continuing.

## Design memory

Read existing Design memory before adding a token, component variant, state pattern, or motion rule. Prefer `DESIGN.md`, project instructions, token files, component documentation, nearby screens, accepted concepts, and saved decision records in that order of local authority.

Maintain one compact snapshot when the work establishes or changes a reusable system:

```text
Design memory
- surface register and route owner:
- DirectionContract revision:
- active Modifier deltas:
- composition fingerprint:
- comparison set and divergence evidence:
- palette and type roles:
- spacing, radius, and depth model:
- component and state families:
- media and motion rules:
- explicit refusals:
```

Include nearby screens and accepted concepts in the comparison set when they are visible in the current task or project. Preserve the same system only with explicit continuity evidence; otherwise a new theme or substantial redesign must not inherit an unrelated composition fingerprint. Reuse the snapshot across slices. Do not silently invent a second system for a new screen or breakpoint.

## Artifact target and maturity

Declare one artifact target before implementation and keep its claim within the evidence actually produced.

| Artifact target | Allowed claim | Matching evidence | Promotion boundary |
| --- | --- | --- | --- |
| recipe | A non-runnable bundle of direction metadata that another implementation may follow. | A complete decision record with inputs, constraints, and acceptance checks. | Do not claim visual or behavioral execution. |
| preview | A visual evidence artifact without behavior claims. | A rendered image or inspected visual artifact with its scenario and viewport. | Do not claim executable workflow behavior. |
| runnable starter | An executable skeleton with its declared basic interaction. | A named entrypoint, successful start command, and behavioral evidence for that basic interaction. | Do not imply complete acceptance or reusable packaging. |
| tested golden | A stable named reference scenario. | A named scenario whose applicable acceptance checks have runtime and rendered evidence. | Do not generalize beyond the named scenario and recorded environment. |
| template | A reusable artifact with an explicit replacement boundary. | A manifest, entrypoint, replaceable-input contract, origin/license record, smoke test, and rendered evidence. | Do not claim production fitness for untested consumers or inputs. |

## Maturity claim rules

Do not promote an artifact without matching evidence.

A recipe or preview must not be described as a production template.

- Name the target and maturity in the implementation record and final report.
- Promote only after the additional requirements for the next maturity are present and verified.
- Downgrade the claim when a required command, scenario, asset, or environment cannot be exercised.
- Treat saved files as existence evidence only; existence does not prove behavior, rendering, reuse, or production fitness.

## Implementation inventory

Inventory the exact material needed to implement the functional and visual contracts:

- routes, entrypoints, commands, package boundaries, and existing components
- selected website archetype, route/page scope, ordered content beats, region geometry, state progression, scroll model, handoffs, proof/conversion rhythm, and many-to-many media coverage
- user-visible copy, actions, state labels, proof strings, and allowed additions
- workflow transitions, data sources, permissions, failures, recovery, and irreversible consequences
- palette, type, spacing, depth, container, icon, media, and motion roles
- selected, disabled, loading, empty, error, success, and permission states
- declared target devices and input model, long-text, localization, reduced-motion, and performance constraints
- asset provenance, authorization, dimensions, semantic role, fallback, and replacement boundary

Record an unknown as an unknown. Do not fill a missing contract with decorative UI or invented product behavior.

## Locked/adaptable rules

Apply these rules in order:

1. Preserve the role and relationship of every Locked decision.
2. Use Adaptable mechanics to satisfy real platform, accessibility, data, and performance constraints.
3. Keep optional Modifier decisions subordinate to the active route and `DirectionContract`.
4. Change one contract field only when the evidence names that field and the owning decision.
5. Escalate when a required adaptation would change the user mission, primary workflow, content truth, or central visual thesis.

Target-device change is adaptation only while it preserves task priority, first read, proof relationship, and action certainty. Do not introduce a mobile requirement when the accepted scope is explicitly PC-only.

## Change ledger

Use the Change ledger when a material decision changes. The cells describe what to record; replace them with project-specific facts and locators.

| Decision | Baseline | Implemented delta | Evidence | Owner |
| --- | --- | --- | --- | --- |
| direction | Accepted `DirectionContract` fields and revision. | Exact locked or adaptable visual decision that changed. | Before/after rendered locator and contract comparison. | Active product or brand route. |
| workflow | Accepted actor, trigger, transition, outcome, and recovery. | Exact behavior implemented or intentionally deferred. | Behavioral test, trace, or runtime observation locator. | Owning product flow. |
| data | Declared source of truth, state shape, and fallback. | Exact read, write, freshness, or fallback change. | Fixture, query, response, or runtime state locator. | Owning data boundary. |
| target-device | Accepted hierarchy, input model, and task priority across declared devices and viewports. | Exact stacking, condensation, relocation, mechanism change, or removal. | Rendered and behavioral locators for the declared device set. | Active route with direction lock. |
| accessibility | Declared semantics, focus, input, and motion behavior. | Exact correction or known constraint. | Semantic inspection and interaction-check locator. | Implemented component or flow. |

## Parity slices

Use Parity slices to implement and compare one vertical slice at a time. Replace the template cells with the named project behavior and evidence.

For a complete route or website, an implementation slice may cover one region, one state progression, or one handoff, but completion requires an end-to-end parity path across every ordered content beat and required route. Never use a finished first viewport as evidence that the complete experience is implemented. Track interactive regions and handoffs as parity slices, then verify the whole route order, geometry, media graph, continuity, and terminal state.

| Parity slice | Behavior | State | Viewport | Evidence | Status |
| --- | --- | --- | --- | --- | --- |
| primary task | The accepted primary transition and outcome. | Default, active, success, and applicable loading state. | Declared target-device contexts. | Behavioral and rendered locators for the same named scenario. | pending, pass, fail, or verification gap. |
| critical recovery | The accepted failure signal and recovery path. | Error, retained input, retry, and recovered state. | Every viewport where recovery is available. | Triggered failure trace plus rendered error and recovery locators. | pending, pass, fail, or verification gap. |
| permission boundary | The accepted allowed and denied outcomes. | Allowed, denied, disabled, and escalation state as applicable. | Every context where authorization changes the surface. | Behavioral permission result plus rendered state locator. | pending, pass, fail, or verification gap. |

Finish the current slice's applicable checks before expanding its pattern. Do not use completion in one slice as evidence for another.

## Target-device adaptation

Design only for the declared target-device scope. For responsive work, derive each supported viewport from the same task and direction contract. For explicit PC-only work, define the intended desktop width range and inputs and do not add mobile layout obligations. Preserve the primary workflow, first read, proof object, action placement logic, content truth, and state clarity while adapting mechanics.

- Define what stays visible, moves, stacks, condenses, scrolls, or becomes a disclosed control.
- Test real long text, localized expansion, sparse data, dense data, and the declared pointer, wheel/trackpad, keyboard, touch, or spatial inputs.
- Replace motion or spatial effects when performance or reduced motion requires it; preserve the communicated relationship.
- Record a change in the ledger when a target device or breakpoint changes hierarchy rather than mechanics.

## Final parity note

End implementation work with a compact record:

```text
Artifact target and maturity:
Functional delta verified:
Locked decisions preserved:
Adaptable decisions used:
Visible or behavioral deviations and owner:
Behavioral and rendered evidence locators:
Remaining verification gap:
```

Keep the note claim-specific. Refer to evidence locators; do not replace missing evidence with confidence or inference.
